SysML v2 · MBSE · VESSE

System-as-Code.
AI can finally read your architecture.

SysML v2 didn't just modernize the model — it made it machine-readable. Textual notation, a standard API, KerML semantics: your architecture is now something an agent can parse, validate, and extend. In the AI age, that changes everything. System-as-Code is the paradigm that makes VESSE workflows truly AI-native.

part def BrakeECU :> ElectronicControlUnit { attribute asilLevel : SafetyLevel = ASIL_D; port canBus : CANInterface; } // one model a tool and an LLM can both read, diff, and generate

The v1 → v2 Inflection

Diagrams can't talk to AI. Code can.

SysML v1 described systems with diagrams locked inside proprietary tools. SysML v2 adds a formal textual notation, a standard API, and the KerML metamodel — so models behave like source code the rest of the toolchain can act on.

SysML v1
  • Diagram-first. Meaning lives in pictures drawn by hand.
  • Proprietary lock-in. Models trapped in one vendor's file format.
  • Hard to parse. Brittle XMI exchange; little machine semantics.
  • Model–code drift. Engineers write, then re-write, in code.
  • Boutique silos. Systems models disconnected from software.
SysML v2
  • Textual notation. Models are text — readable, reviewable, writable.
  • Standard API. A REST/Platform API any tool or agent can call.
  • Git-native. Diff, branch, and review models like a pull request.
  • KerML metamodel. Precise semantics LLMs can reason over.
  • One source of truth. Requirements, structure, behavior — unified.
Why it matters here: v1 made AI integration a research project. v2 makes it an API call. Everything downstream in this pitch — agentic analysis, code generation, CI/CD — only becomes practical once the model is code.

The Platform

Model as API. Code on demand.

Syside (Sensmetry) is one of the most SysML v2-native tools available today. The Modeler + Automator pattern it demonstrates applies to any SysML v2-conformant toolchain — including Cameo and Rhapsody as they add v2 support.

Authoring

Modeler

A fast, feature-rich SysML v2 environment for engineers.

  • Textual editing with autocomplete and refactoring.
  • Live validation as you type.
  • Automatic diagram generation from the text.
  • Headless CLI for batch and pipeline use.
Automation

Automator

A Python API for programmatic SysML v2 model access.

  • Generate reports, docs, and diagrams on demand.
  • Drive CI/CD checks against the model.
  • Sync requirements with DOORS and PLM.
  • Template-driven code and test generation.
The PM read: Ford doesn't need to build a modeler, and doesn't need to abandon Cameo or Rhapsody. The leverage is in the Automator's Python + SysML v2 API surface — a toolchain-agnostic integration bus that plugs into whatever SysML v2-conformant environment Ford standardizes on.

VESSE in Practice

What System-as-Code makes real.

Standard Use Cases mapped to a concrete VESSE workflow and the KPI it moves.

Automatic asset generation from the model UC 1

Generate documentation, interface diagrams, and review reports in any format directly from the model, kept in sync through the Python interface.

  • VESSE application: auto-produce ECU interface specs, E/E architecture reports, and design-review packs the moment the model changes — no manual redraw.
  • KPI: engineering hours saved; documentation always current with the model.
Model simulation, analysis & validation UC 2

Execute and validate system models from the design phase — run model checks and simulate behavior before implementation.

  • VESSE application: catch interface mismatches and timing/latency violations at design time, shifting defect discovery left of MIL/SIL/HIL.
  • KPI: Defect Detection Rate (DDR); cost avoided by not finding defects at HIL or in-market.
Integrating multiple engineering tools & workflows UC 3

Connect existing tools through Python interfaces and the SysML v2 API — PLM, requirements management, and CAD — while keeping the model consistent.

  • VESSE application: a live bridge between DOORS requirements, Cameo models, and AUTOSAR repositories so trace links stay intact across the toolchain.
  • KPI: traceability compliance; hours lost to manual cross-referencing eliminated.
Collaborative model editing UC 4

Work on models simultaneously with real-time collaboration and version control — track changes, manage conflicts, maintain consistency across the team.

  • VESSE application: model changes reviewed like code — branch, diff, merge — across distributed electrical, software, and systems teams.
  • KPI: review cycle time; fewer integration conflicts late in the V-model.
Structured artifact generation for toolchain hand-off UC 5

Generate structured interface specifications and configuration inputs from the model using template engines via the Automator — not to bypass AUTOSAR toolchains, but to reduce the manual burden of feeding them.

  • VESSE application: produce ARXML interface descriptions, software component port definitions, and signal lists that feed directly into DaVinci or EB tresos — eliminating the manual transcription step between model and AUTOSAR toolchain. SIL/HIL test case stubs are a parallel output where data quality supports it.
  • KPI: hours saved at model-to-toolchain handoff; reduction in interface definition errors caught late in integration.
Embedding DSLs into SysML v2 models UC 6

Create and integrate domain-specific languages within models to express specialized engineering concepts while staying compatible with the SysML v2 standard.

  • VESSE application: encode Ford-specific E/E, network, and safety semantics as first-class DSL constructs — capturing institutional knowledge in the model itself.
  • KPI: reuse across programs; reduced onboarding time for new engineers.
